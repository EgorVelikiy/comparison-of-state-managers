# React State Managers Comparison

Тестовое React-приложение для сравнительного анализа инструментов управления состоянием в рамках выпускной квалификационной работы.

## Технологический стек

- **React 18** - библиотека для создания пользовательских интерфейсов
- **TypeScript** - типизированный надмножество JavaScript
- **Vite** - инструмент сборки и разработки
- **State Managers:**
  - Redux Toolkit
  - MobX
  - Recoil
  - Valtio

## Установка и запуск

### Полная установка (клиент + сервер)

1. Установите зависимости для клиента и сервера:
```bash
npm run install:all
```

Или вручную:
```bash
# Установка зависимостей клиента
npm install

# Установка зависимостей сервера
cd server
npm install
cd ..
```

### Запуск приложения

**Важно:** Для работы приложения необходимо запустить как сервер, так и клиент.

1. **Запустите Express сервер** (в отдельном терминале):
```bash
npm run server:dev
# или
cd server && npm run dev
```

Сервер будет доступен на http://localhost:3001

2. **Запустите клиентское приложение**:
```bash
npm run dev
```

Приложение будет доступно на http://localhost:5173

### Production сборка

1. Соберите клиент:
```bash
npm run build
```

2. Соберите сервер:
```bash
npm run server:build
npm run server:start
```

## Функциональность

Приложение реализует data-intensive dashboard с следующими возможностями:

1. **Отображение списка элементов** (по умолчанию 1000 элементов)
2. **Операции:**
   - Массовое обновление поля/полей у всех элементов
   - Локальное обновление одного элемента
   - Фильтрация списка по поиску, категории и статусу
   - Выбор элемента и отображение детальной информации
   - Асинхронная загрузка данных (симулируется через задержку)

3. **Derived State:**
   - Общее количество элементов
   - Количество отфильтрованных элементов
   - Среднее значение всех элементов
   - Количество активных элементов
   - Количество неактивных элементов

## Особенности реализации

### Архитектурные решения

- **Идентичная структура компонентов** для всех реализаций state managers
- **Изоляция логики state management** в отдельной папке `state/`
- **Общие UI компоненты** в папке `components/shared/`
- **Отсутствие оптимизаций** (memo, useCallback, virtualization) для чистоты сравнения

### Анализ производительности

- **Логирование ререндеров** - счетчик ререндеров для каждого компонента
- **React Profiler** - приложение пригодно для анализа через React Profiler
- **Lighthouse** - приложение готово для анализа производительности

### Redux Toolkit

- Использование `createSlice` для создания reducer и actions
- `createAsyncThunk` для асинхронных операций
- `createSelector` для derived state
- Типизированные хуки `useAppDispatch` и `useAppSelector`

### MobX

- Класс-based store с `makeAutoObservable`
- Computed values для derived state
- `observer` HOC для реактивности компонентов
- `runInAction` для обновления состояния в async функциях

### Recoil

- Atoms для базового состояния
- Selectors для вычисляемых значений и derived state
- Отдельные функции-actions для обновления состояния

### Valtio

- Proxy-based состояние через `proxy()`
- `derive` для вычисляемых значений
- `useSnapshot` для реактивной подписки в компонентах

## Использование

1. Выберите state manager из выпадающего списка в header
2. Используйте фильтры для поиска и фильтрации элементов
3. Нажмите на элемент для просмотра детальной информации
4. Используйте кнопки для массового обновления и загрузки данных
5. Наблюдайте за счетчиками ререндеров в правом нижнем углу

## Анализ производительности

Приложение разработано для проведения сравнительного анализа производительности различных state managers:

1. Используйте React Profiler для измерения времени рендеринга
2. Проверяйте счетчики ререндеров компонентов
3. Анализируйте производительность через Lighthouse
4. Сравнивайте результаты для разных state managers при одинаковых операциях

## API Сервер

Проект включает Express сервер для работы с данными. Подробная документация по API доступна в [server/README.md](server/README.md).

### Основные endpoints:

- `GET /api/items` - Получение списка элементов
- `GET /api/items/:id` - Получение элемента по ID
- `PUT /api/items/:id` - Обновление элемента
- `POST /api/items/bulk` - Массовое обновление элементов
- `GET /health` - Проверка состояния сервера

### Конфигурация API

URL API сервера настраивается через переменную окружения:
- Создайте файл `.env` в корне проекта
- Добавьте: `VITE_API_URL=http://localhost:3001/api`
- По умолчанию используется `http://localhost:3001/api`

## Важные замечания

- **Не используются оптимизации** (memo, useCallback) для чистоты сравнения
- **Идентичные сценарии обновления** состояния во всех реализациях
- **Минимальные стили** без UI-библиотек
- **Реальный API** - приложение использует Express сервер для асинхронных запросов
- **Код готов для анализа** в рамках выпускной квалификационной работы
